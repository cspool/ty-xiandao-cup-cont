为什么 decode 慢到"算力闲置"、双缓冲还救不了?

两个高频困惑,一次讲透:①prefill / decode 的权重 IO、KV-cache IO、计算各占多少时间? ②为什么双缓冲做了没效果、VGPR/LDS 也不高、并发还是上不去?本文所有结论都来自我在 DCU 上跑的真实微基准。

实测 HBM 峰值 1206 GB/s 实测 bf16 算力峰值 ≈395 TFLOPS roofline 拐点 ≈327 FLOP/byte wave64 · gfx936
先给两句话结论(后面用数据证)
  • 问题①:decode 是权重 IO 说了算——实测算力只用了峰值的 0.3%,时间几乎全在"把权重从显存搬进来";prefill 相反,是计算说了算(权重被成百上千个 token 复用摊薄)。
  • 问题②:你的 GEMV 本来就贴着带宽墙了(实测无双缓冲已达 HBM 峰值的 92–101%)。双缓冲/占用率是"藏延迟"的工具,可你不是被延迟卡住、而是被带宽卡住——所以做了没用。唯一出路是减少要搬的字节(量化)。

1prefill vs decode:三种资源各占多少时间

先分清"三种资源"

一层 Transformer 的耗时,拆成三个互相独立的硬件资源:

  • 权重 IO把模型权重从 HBM 搬进计算单元。字节数固定(权重就那么大),与一次处理多少 token 无关。
  • KV-cache IO读写注意力的 KV 缓存。字节数随上下文长度增长。
  • 计算张量/向量单元真正做的乘加(FLOPs)。随处理的 token 数增长。

到底哪个成为瓶颈,由一个量决定:算术强度(Arithmetic Intensity)= FLOPs ÷ 搬运字节数,即"每搬一个字节,能顺带算多少次"。 它和硬件的拐点 = 算力峰值 ÷ 带宽峰值 比较:低于拐点=访存受限,高于=计算受限。 我实测这台 DCU:算力峰值 ≈395 TFLOPS、HBM ≈1206 GB/s → 拐点 ≈ 327 FLOP/byte。

把实测换算到整个 27B 模型的一次 decode

Qwen3.5-27B 全部权重 ≈ 54 GB(bf16)。按实测 decode 带宽 ~1200 GB/s:

  • 权重 IO54 GB ÷ 1.2 TB/s ≈ 45 ms / token —— 和我们实测的 bf16 baseline TPOT(≈49 ms)几乎吻合。
  • 计算每 token 约 2×27e9 = 54 GFLOP,÷ 395 TFLOPS ≈ 0.14 ms —— 只占 0.3%,算力单元 99.7% 时间在等权重。
  • KV IO这是混合架构(64 层里仅 16 层 full-attn 有 KV,48 层线性注意力是定长状态)。每 context-token 约 64 KB → 8K 上下文 ≈ 0.4 ms、32K ≈ 1.7 ms。相对 45ms 的权重 IO 仍是小头,但长档时占比上升。

把 prefill 也展开:投影 GEMM 是 O(S),注意力是 O(S²)

prefill 一次吞下整段 prompt(S 个 token),它的"计算"其实是两种增长速度不同的东西叠加:

  • 投影 GEMMQKV / O / gate_up / down 这些线性层,FLOPs = S·K·N,随 token 数线性增长 = O(S)。它们是标准的大矩阵乘,计算受限、贴着算力峰值。
  • 注意力QKᵀ 和 softmax·V 要让每个 token 看所有 token,FLOPs = S²·d,随长度平方增长 = O(S²)。短档不起眼,长档会反超一切。

实测把两条定律钉死了:S 每翻一倍,GEMM ×2(O(S) 铁证),注意力 ×3.2–4(O(S²) 铁证)。 attn/GEMM 从 0.61 一路涨到 6.50——注意力在 S≈1K 后就反超投影,长档时是绝对主力。这正是 prefill 优化的真正战场。

对上真实 TTFT:长档确实超线性

我们实测的 prefill 首 token 延迟(TTFT):8K = 2790ms、16K = 5371ms、32K = 11423ms。

  • 8K→16K:5371/2790 = 1.93×(token 翻倍,近似线性——这个长度还是投影 GEMM 为主)。
  • 16K→32K:11423/5371 = 2.13×(token 翻倍却涨了 2.13 倍——超线性开始显现,正是 O(S²) 注意力在抬头)。

救星是混合架构:这个模型 64 层里只有 16 层是 full-attention(O(S²)),另外 48 层是 GDN 线性注意力(O(S))。 正因如此,它的 TTFT 增长比"纯注意力模型"温和得多——否则 32K 早就爆炸了。

一句话对比 decode 与 prefill(优化方向截然相反)

decode = 访存受限,瓶颈是权重带宽 → 减字节(量化); prefill = 计算受限,投影 GEMM 已到算力峰值(≈397 TFLOPS)几乎无空间,瓶颈随长度转移到注意力 O(S²) → flash-attention(S×S 不落显存)。 注意:量化对 prefill 帮助很小(权重已被 S 个 token 摊薄,反而多一步反量化),这正是我们 down 量化拖累 16–32K 长档的原因。

DCU / HIP 视角

这个结论是硬件无关的,NVIDIA / DCU 一样:decode 访存受限、prefill 计算受限。落到 gfx936 上: decode 的 GEMV 用手写 HIP 核榨带宽(__builtin_nontemporal_load 流式读、wave64 一行、__shfl_xor 规约); prefill 的大 GEMM 交给 rocBLAS / hipBLASLt 上 Matrix Core(对标 cuBLAS + Tensor Core)。诊断带宽用 rocprof / omniperf 看 MemUnitBusy,对标 ncu 的 DRAM Throughput。

2为什么双缓冲没效果、并发也上不去

先把工具的作用说准:双缓冲(软件流水)和高占用率,都是用来"藏内存延迟"的。 它们能不能帮上忙,取决于你到底被什么卡住——被延迟卡住,它们有用;被带宽卡住,它们一点用没有。

延迟受限 vs 带宽受限:两码事

延迟受限(latency-bound)

波太少,一个 load 发出去要等几百周期才回来,期间计算单元干等。此时双缓冲/多波并发能把等待藏起来 → 提速明显。

带宽受限(bandwidth-bound)

HBM 已经在满速吐数据,瓶颈是"管子粗细"不是"等回信"。此时再多波、再双缓冲,也不能让管子变粗 → 提速为零。

那 VGPR=58 / LDS=32 不高,为什么反而是"双缓冲没用"的原因?

这正是关键、也最反直觉的一点:

  1. VGPR/LDS 低 = 占用率高 = 常驻的 wave 很多。GEMV 每行一个 wave,一个 gate_up 就是 34816 个 wave 排队; 资源又省(58 VGPR),每个 SIMD 上能同时驻留满额的 wave。
  2. 占用率高,硬件自己就把内存延迟藏掉了。GPU 的看家本领是"一个 wave 等 load 时,立刻切到另一个 wave 算"。 常驻 wave 一多,总有 wave 可算,访存延迟被 wave 切换天然覆盖——这件事双缓冲也想做,可硬件已经做完了。
  3. 所以你手写的双缓冲是"重复劳动",自然没有增量。软件双缓冲真正救得了的,是占用率低的场景(VGPR 爆了、每 SIMD 只剩一两个 wave,切无可切,只能靠软件流水提前预取)。你 VGPR 才 58、占用率高,轮不到它出场。
  4. 并发"上不去"的错觉:不是并发不够,而是并发早已过剩。几千个 wave 抢的是同一条 HBM 通道,一旦通道满速,再加并发只是排更长的队 → 带宽不涨、延迟反而略增。共享资源(带宽)的瓶颈,加并行度是解不开的。
"只做了热点算子" —— 另一个让你觉得"没提速"的坑(Amdahl)

就算把热点 GEMV 榨到峰值,端到端 TPOT 也可能纹丝不动,因为 decode 每个 token 要跑几十个算子: RMSNorm、rope、逐元素、sampling(大词表 softmax/topk)、还有算子之间的 kernel launch 开销与同步空隙。

  • 热点算子从 100% 峰值再优化,收益趋于 0;而没碰的那些小算子 + 每次 launch 的几微秒 × 几十次 × 每 token 累积起来可能才是新的大头(用图捕获消 launch、算子融合减往返即可,这里不展开)。

怎么快速自查"我到底被什么卡住"

  1. 算 roofline:测你核的达成带宽(字节÷时间),除以 HBM 峰值。≥85% 就是带宽墙——别再碰双缓冲/占用率,去减字节。
  2. 看 profiler(rocprof / omniperf):MemUnitBusy≈100% 且 VALUBusy 很低 → 访存受限;若 MemUnitStalled 高但 MemUnitBusy 不满 → 才是延迟受限(这时双缓冲/提占用才有用)。
  3. 看 VGPR/LDS 判断占用率天花板:低(如 58/32)= 占用率不是限制 → 更加说明"提占用/双缓冲无空间"。
结论:带宽墙下,方向只有"减字节"

被带宽卡死时,双缓冲、提占用率都不在有效范围内(你已经在墙上了)。真正的方向是减少要搬的字节——权重 int8/int4 量化(读一半字节、时间近乎减半),这就是我们把 TPOT 从 ~49ms 压到 ~40ms 的主武器;其余靠对齐流式读提有效带宽、图融合减 launch,不再展开。
