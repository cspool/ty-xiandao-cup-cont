LECTURE 02

Torch-Pruning：结构化剪枝

给模型“做减法”而不留窟窿 —— 用 DepGraph 自动找出“必须一起剪的一组”，真正减体积、降显存、提速度

上一讲的 FP16 是“把每个数记得更省”，但模型该算的还是全算了一遍。这一讲换一种思路：干脆把不重要的部分整块删掉—— 少算、少存、少占显存。难点在于神经网络的“枝条”互相牵连，剪错会让模型直接崩掉。 torch-pruning 1.6.1 的核心武器 DepGraph 就是来解决这个牵连问题的。

§0　上一讲回顾 & 思考题全解

上一讲要点速览
  • ONNX = 把模型翻译成框架无关的“计算图”（节点=算子、边=张量），是一切图级优化与部署引擎的统一入口。
  • onnxsim：常量折叠 + 算子融合 + 去冗余，只改结构不改数值。
  • onnxconverter-common（FP16）：每个数 4 字节 → 2 字节，体积/显存约减半；敏感算子用 op_block_list 保 FP32，或 auto_mixed_precision 自动兜底。
  • 固定顺序：导出 → simplify → FP16 → 交给引擎编译；U/V 靠它拿分，W（精度）要在验证集实测守住。
Q1 解答：为什么训练爱用 BF16、推理爱用 FP16 关键在 16 位预算怎么切：FP16 = 1 符号 + 5 指数 + 10 尾数；BF16 = 1 符号 + 8 指数 + 7 尾数。指数位决定范围，尾数位决定精度。
训练用 BF16：反向传播里梯度的动态范围极大，很小的梯度在 FP16（范围仅 ±65504）里会下溢成 0、很大的会溢出成 inf，得靠 loss scaling 兜。BF16 保留了和 FP32 一样的 8 位指数 → 范围几乎不缩，天然不易溢出，训练稳定；尾数糙一点没关系，SGD 本身对噪声鲁棒。
推理用 FP16：权重已训好、数值落在有界的良性区间，更在意每个数的有效数字。FP16 的 10 位尾数 ≈ 3~4 位十进制精度（BF16 只有约 2~3 位），数值保真更好；且 FP16 被各推理硬件与引擎（Tensor Core、MIGraphX 等）原生支持。一句话：训练要范围（BF16），推理要精度（FP16）。
Q2 解答：Softmax→ReduceMean 分支转 FP16 后出 NaN 原因：这条分支全是“易爆”操作。Softmax 里的 exp() 对稍大的 logit 就会算出超过 FP16 上限（±65504）的值 → inf；ReduceMean/ReduceSum 把很多数累加，FP16 尾数少，累加既容易溢出又丢精度。一旦出现 inf，后续 inf−inf 或 inf/inf 就变 NaN 并污染整条链路。
修法一（手工）：把这些算子保留为 FP32——convert_float_to_float16(model, op_block_list=["Softmax","ReduceMean","ReduceSum"])，或用 node_block_list 精确点名那几个节点；再配合 min_positive_val/max_finite_val 钳制数值。
修法二（自动）：用 auto_mixed_precision.auto_convert_mixed_precision，给一组校验输入和容差函数，它会自动定位“一转就崩/掉精度”的算子并保留 FP32。
Q3 解答：FP16 对“模型大小”和“显存”的改善为何不完全一样 模型大小：主要是权重，FP32→FP16 每个数 4→2 字节，文件≈ 减半（除非有些静态数据/查表没转）。
显存：推理峰值显存 = 权重 + 激活（中间特征图） + 输入输出缓冲 + kernel 的临时 workspace + 运行时固定开销。权重和激活转 FP16 都减半，但：① 若 keep_io_types=True，输入输出仍是 FP32；② block 掉的算子其激活仍是 FP32；③ workspace 与运行时固定开销不随精度缩小。所以显存改善很可观，但通常达不到精确的一半。这也提醒：评分 U 的显存那 20 分必须实测推理峰值显存，不能拿文件大小代替。
Q4 解答：只用“导出+simplify+FP16”的 pangu-weather 流程 checklist
  1. 测基线：跑通全量 FP32，记录 模型大小（目录+权重）、推理峰值显存、单样本推理时长（按规定计时段）、各气象指标 RMSE/ACC——这些是所有比值的分母。
  2. 导出：model.eval() → torch.onnx.export（opset≥17，必要时设 dynamic_axes）；checker 校验 + ORT 与 PyTorch 数值对齐（allclose）。
  3. 简化：onnxsim.simplify，assert ok，再对齐一次数值。
  4. 转 FP16：先 keep_io_types=True 全转，在验证集实测 RMSE/ACC。
  5. 救精度：哪个指标掉/出 NaN，就把 LayerNorm/Softmax/ReduceMean 等加进 op_block_list，或上自动混合精度。
  6. 上卡编译：用 MIGraphX 编译，实测推理时长与峰值显存。
验证指标 checklist：□ 体积↓ □ 峰值显存↓（实测） □ 单样本时长↓ □ 每个气象指标 RMSE/ACC 相对原模型未明显变差。

§1　非结构化 vs 结构化：为什么“剪得整齐”才真省

开场类比：删字 vs 删段 给一篇文章瘦身有两种删法。非结构化像“在全文里东划掉一个字、西划掉一个字”——划掉的地方多，但排版还得给每个空位留位置，页数没真减少。 结构化像“整段整段地删”——删完直接少几页。神经网络也一样：零散地把单个权重置零（非结构化），矩阵形状没变、通用硬件照样按稠密矩阵算，省不了多少； 整通道/整滤波器/整层地删（结构化），矩阵真的变小了，算量、权重、显存才会同步下降。

两者的本质区别

  • 非结构化剪枝（unstructured）：按权重大小等标准，把单个权重置零，得到一个稀疏但形状不变的权重矩阵。灵活、精度损失小，但要真加速得靠专门的稀疏算子/硬件支持，否则在国产卡这类通用加速器上“看着稀疏、算得不快、占的显存也没少”。
  • 结构化剪枝（structured）：成组地删除整个通道 / 卷积核 / 注意力头 / 神经元，剩下的仍是一个更小的稠密权重。直接缩小矩阵维度 → 参数量、MACs、显存、延迟全都真实下降，无需特殊硬件。
为什么本赛题必须用结构化剪枝 评分的 U（体积/显存）和 V（速度）要的是“在国产加速卡上真实变小、变快”。非结构化的稀疏矩阵在通用硬件上既不省显存也不提速—— 所以 torch-pruning 主打的“结构化剪枝”才是对分数有效的那把刀。

§2　牵连问题与 DepGraph：剪一刀，牵动一片

开场类比：拆水管不能只拆一头 假设你想拆掉一根水管，那它两端接的接口、阀门、下游连着的管子都得跟着处理，否则整条管路就漏了。 神经网络里，删掉某层的一个输出通道，下一层对应的输入通道就悬空了，中间的 BN、残差相加、拼接也都跟着对不上。 所以结构化剪枝的真正难点不是“删谁”，而是“删一个，必须连带删哪些才能保持整张图自洽”。

四类典型依赖：必须“成组同删”

上面 §1 图 (b) 的教训就是：耦合的两层如果删得不一致，模型就废了。下图列出几种必须同步剪的依赖结构：

DepGraph 在做的事：自动找出“最小可删单元 Group” DepGraph（Dependency Graph，依赖图）会扫描整个模型，建一张记录“谁和谁的通道是绑定的”的依赖图，然后把所有相互牵连、必须一起剪的层归成一个 Group（组）。 Group 就是结构化剪枝里“能被安全删除的最小整体”——你只要说“想剪某层的第 3、7、9 号通道”，DepGraph 会自动把这一组里所有耦合层的对应通道一并剪掉，并修好 BN、残差、拼接，保证剪完图仍然能跑。 这正是 torch-pruning 相对“手动剪”的核心价值：把易错的牵连关系自动算对。

它适用于几乎所有主流结构

§3　剪谁、剪多少、怎么剪：重要性度量与策略

开场类比：精简团队先看“贡献度” 要给团队精简编制，你会先评估每个人的贡献，再裁掉贡献最低的。剪枝同理：先给每个通道/滤波器算一个重要性分数，把分数最低的那批删掉。

① 重要性度量：怎么判断一个通道重不重要

  • 幅值范数（L1/L2 norm）：通道权重的绝对值之和/平方和越小，说明它输出的贡献越弱，越该剪。简单、稳、最常用。
  • Taylor / 梯度类：用一阶泰勒展开估计“删掉这个通道，损失会涨多少”，需要一点校验数据跑反向。更准，但更重。
  • 随机 / 其他：作基线对照。

在 torch-pruning 里，这些被封装成 importance 对象（如 MagnitudeImportance、TaylorImportance、GroupNormImportance），而且是按 Group 整体打分的——因为要删就是一整组一起删。

② 剪枝率：删多少

用 pruning_ratio 控制整体删除比例（如 0.5 = 目标砍掉一半通道），也可对不同层设不同比例。剪得越狠，U/V 越高，但 W（精度）越危险——这正是赛题里要找的 Pareto 平衡点。

③ 一次性 vs 迭代式

  • 一次性（one-shot）：一刀剪到目标比例，再微调。快，但剪太狠时精度难回。
  • 迭代式（iterative）：剪一点 → 微调 → 再剪一点 → 再微调，分多步逼近目标。慢，但同样压缩率下精度通常更好。torch-pruning 支持用 pruner.step() 一步步推进。
一条铁律：剪完几乎一定要微调 删通道会直接抹掉一部分已学到的表示，精度通常立刻下滑。剪枝 + 微调（fine-tune）是一套连招，不能只剪不练。微调让剩下的通道“接管”被删通道的职责，把精度拉回来——这一步决定了你的 W 能不能守住。

§4　Torch-Pruning API：剪枝五步走

高层 Pruner 的标准套路 torch-pruning 把“建依赖图 + 按 Group 打分 + 全局排序 + 迭代剪 + 改结构”都封装进一个 pruner。你只需提供： ① 模型与样例输入（用来建 DepGraph）；② 重要性度量；③ 剪枝率；④ 必须保护、不能剪的层（最关键是输出层）。
PYTHON · 结构化剪枝标准流程
import torch, torch_pruning as tp
import torchvision.models as tvm

model = tvm.resnet18(weights="DEFAULT").eval()
example = torch.randn(1, 3, 224, 224)

# ① 重要性度量：按 Group 的 L2 范数打分
imp = tp.importance.GroupNormImportance(p=2)

# ② 保护层：分类头不能剪（剪了输出维度就变了！）
ignored = [m for m in model.modules() if isinstance(m, torch.nn.Linear) and m.out_features == 1000]

# ③ 构造 Pruner：内部自动 build DepGraph、分组、全局排序
pruner = tp.MetaPruner(
    model, example,
    importance=imp,
    pruning_ratio=0.5,            # 目标：砍掉约一半通道
    ignored_layers=ignored,        # 输出层 / 不想剪的层
    iterative_steps=5,            # 分 5 步迭代剪，更稳
)

base_macs, base_params = tp.utils.count_ops_and_params(model, example)

# ④ 迭代：每 step 剪一点，然后微调几个 epoch
for i in range(5):
    pruner.step()                      # 真正修改模型结构（同组通道一并删）
    macs, params = tp.utils.count_ops_and_params(model, example)
    print(f"step {i}: params={params/1e6:.2f}M  MACs={macs/1e9:.2f}G")
    # finetune(model, train_loader, epochs=1)   # ⑤ 每步后微调回补精度
剪完得到的是“真小”的模型 pruner.step() 之后，模型里的卷积/线性层的 out_channels/in_channels 是真的变小了，不是打掩码。 所以可以直接 torch.save 存权重、或导出 ONNX——文件、显存、推理时长都会真实下降。这正好能接上第一讲：剪完再导出 ONNX → simplify → FP16，收益叠加。

§5　完整示例：ResNet18 剪一半，看收益与代价

目标 把 ResNet18 用结构化剪枝砍掉约一半通道，量化对比参数量、MACs（计算量）、精度在“剪枝前 / 剪枝后 / 微调后”三个时刻的变化，建立“剪多少、掉多少、能回补多少”的直觉。
PYTHON · 剪枝 + 微调 + 评测
# 接上一段：pruner 已迭代剪到目标比例
pruned_macs, pruned_params = tp.utils.count_ops_and_params(model, example)
print(f"参数量: {base_params/1e6:.1f}M -> {pruned_params/1e6:.1f}M")
print(f"MACs : {base_macs/1e9:.2f}G -> {pruned_macs/1e9:.2f}G")

# 微调：用训练集把精度拉回来（剪枝后这步几乎必做）
opt = torch.optim.SGD(model.parameters(), lr=1e-3, momentum=0.9)
for epoch in range(5):
    model.train()
    for x, y in train_loader:
        opt.zero_grad()
        loss = torch.nn.functional.cross_entropy(model(x), y)

# 存权重 / 导出 ONNX（结构已真实变小）
torch.save(model.state_dict(), "r18_pruned.pth")
你会看到的典型结果（量级感）
  • 参数量 / MACs：pruning_ratio=0.5 大致让两者降到原来的 1/4 ~ 1/2（通道两端都缩，参数约按平方关系下降）；
  • 精度：刚剪完可能掉好几个点甚至更多；微调几个 epoch 后通常能回到接近原始精度，剩下零点几到一两个点的差距；
  • 结论：剪枝是“用一点精度 + 一段微调，换实打实的体积/显存/速度”，且收益比 FP16 更大、更可调。

§6　易错点 · 一图流 · 赛题落点

⚠️ 易错点汇总

  • 剪了输出层：分类头/回归头维度被改，输出对不上 → 一定要放进 ignored_layers。
  • 残差 / 拼接处手动剪：最容易剪出维度不匹配的 bug——这正是要用 DepGraph 自动分组的原因，别手动绕过它。
  • 只剪不微调：精度塌了还以为方法不行；微调是连招的后半段。
  • 一刀剪太狠：剪枝率过高、又不迭代，精度回不来；改用 iterative_steps 分步剪。
  • 分组卷积 / 注意力头：有整除约束（如剪完通道数要能被组数/头数整除），torch-pruning 会处理，但要理解约束在哪，别设出无法整除的比例。
  • 拿“掩码稀疏”冒充结构化：置零不改形状＝不省显存不提速，对评分无效。
🎯 一图流小结 结构化剪枝 = 整通道地删 → 真小、真快、真省显存（非结构化只是置零，通用硬件不买账）。
难点是“牵连”——删一个通道会波及下游输入、BN、残差、拼接；DepGraph 自动把耦合层归成 Group，保证成组同删、图仍自洽。
流程：定重要性(L1/L2/Taylor) → 设剪枝率 → 保护输出层 → 迭代剪 + 微调 → 存/导出。剪枝率越高 U/V 越高、W 越险，找 Pareto 点。
🔗 赛题落点：在 pangu-weather 上怎么剪
  • 剪哪里：pangu-weather 是 3D Transformer，重点剪 MLP（FFN）的隐藏维和注意力的投影通道/头（见 §2 那张图的 Transformer 分支），这些是参数与计算的大头。
  • U / V（≈60 分）：结构化剪枝直接缩小权重矩阵 → 体积↓、显存↓、单样本推理时长↓，收益比纯 FP16 更大、更可调。
  • W（精度 40 分）：剪完务必用 ERA5 训练/验证集微调或蒸馏回补，在验证集盯住各气象指标的 RMSE/ACC，按 Pareto 选剪枝率。
  • 叠加：剪枝(去结构冗余) → 导出 ONNX → simplify → FP16(压精度位宽) 串起来，收益相乘；下一讲的张量分解再压“低秩冗余”，三招组合。

✏️ 思考题（下一讲开头给出完整解答）

为什么“非结构化剪枝把 90% 权重置零”在国产加速卡这类通用硬件上，往往既不省显存也不提速？而结构化剪枝即使只删 30% 通道，却能实打实地降体积/显存/延迟？请从“矩阵形状 vs 稀疏掩码”和“硬件怎么算”两个角度说清楚。
残差连接 y = F(x) + x 处，如果你只剪了 F(x) 分支最后一层的输出通道、却没动 shortcut 分支，会发生什么？DepGraph 会如何处理这个 Group？（提示：相加要求两路通道数一致。）
同样要把模型压到“参数减半”，一次性剪枝 vs 迭代式剪枝在最终精度上通常谁更好、为什么？如果你时间很紧，会怎么折中？
在 pangu-weather 上，你打算对 MLP 用 pruning_ratio=0.5、对注意力投影用 0.3，并保护 patch embedding 与最终预测头。请说明：①为什么对不同模块设不同比例；②哪些层必须进 ignored_layers；③剪完到拿到最终 U/V/W 分数之间，你还要做哪几步（含验证指标）。
