LECTURE 01

ONNX 三件套：图简化 + FP16 转换

把模型“翻译成标准流程图”，再精修流程图、换用更省的记账单位 —— onnx · onnxsim · onnxconverter-common

这是“模型轻量化三讲”的第一讲，对应先导杯气象赛题里组委会预装的第一组武器： onnx 1.21.0 / onnxsim 0.6.3 / onnxconverter-common 1.16.0。 这三件套不改变模型“算什么”，只改变模型“怎么被存、怎么被算”——是性价比最高、最该最先做的一步： 几乎不掉精度，就能把体积、显存砍掉一半，并明显提速，直接拿下评分里的 U、V。

🎯 本讲学完你会……

  • 说清 ONNX 到底是什么——一张“算子节点 + 张量边”的计算图，以及为什么部署一定要先走它；
  • 用 torch.onnx.export 把 PyTorch 模型导出成 ONNX，并验证数值一致；
  • 用 onnxsim 做常量折叠 / 算子融合 / 去冗余，把导出时产生的“一团乱麻”精简成干净图；
  • 用 onnxconverter-common 把模型转 FP16，理解它为什么省一半、以及哪些算子会“翻车”要 block 掉；
  • 跑通一个小 CNN 的完整流水线，量化对比体积/显存/数值误差；
  • 知道这一套在 pangu-weather + 国产加速卡(MIGraphX) 上具体怎么落地拿分。

§1　先认识主角：什么是 ONNX，为什么部署绕不开它

开场类比：菜谱 vs 厨师 PyTorch 模型像是“某位厨师的私房做法”——只有这位厨师（PyTorch 运行时）在、用他那套锅具才能做。 而 ONNX 是把这道菜写成一张标准化、人人能照做的菜谱：每一步（算子）、每样食材（张量）都写得清清楚楚。 有了这张标准菜谱，换谁来做都行——ONNX Runtime、TensorRT、国产卡的 MIGraphX……都能照着高效执行。 轻量化的第一步，就是先把模型“翻译”成这张标准菜谱。

ONNX = 一张计算图

ONNX（Open Neural Network Exchange）是一种开放的模型中间表示（IR）。它把一个神经网络描述成一张 有向无环图（DAG）：

  • 节点（node）＝算子（operator），如 Conv、MatMul、Relu、Add，每个节点有固定语义和一个版本号 opset；
  • 边（edge）＝张量（tensor），在节点间流动；其中模型权重是“初始化好的常量张量（initializer）”；
  • 图的输入/输出＝整张图的接口，带名字、形状（可含动态维度）、数据类型。
为什么要先转 ONNX（动机） ① 框架无关：脱离 PyTorch，谁都能加载；② 图级优化的舞台：算子融合、常量折叠、精度转换都在这张静态图上做最方便； ③ 部署引擎的统一入口：几乎所有高性能推理引擎（含本赛题国产卡的 MIGraphX、以及 ONNX Runtime）都把 ONNX 当作标准输入。 一句话：想让模型在目标硬件上又快又省，先把它变成一张能被反复优化的标准图。

导出：torch.onnx.export

导出的本质是“拿一个样例输入，把模型跑一遍（trace），把经过的算子记成图”。所以你必须给一个 dummy 输入：

PYTHON · 导出 ONNX
import torch
model = MyNet().eval()                      # ① 一定先 eval()，固定 BN/Dropout
dummy = torch.randn(1, 3, 224, 224)        # ② 样例输入，决定 trace 路径

torch.onnx.export(
    model, dummy, "model.onnx",
    opset_version=17,                       # ③ 算子集版本，建议 ≥13，越新算子越全
    input_names=["input"], output_names=["output"],
    dynamic_axes={"input": {0: "batch"},     # ④ 让 batch 维可变；不写则形状被写死
                 "output": {0: "batch"}},
)
导出后务必验证两件事 (1) 图合法：onnx.checker.check_model(onnx.load("model.onnx"))； (2) 数值一致：用 ONNX Runtime 跑一遍，和 PyTorch 输出对比， np.allclose(torch_out, ort_out, atol=1e-4)。 导出阶段的“图对不对”是后面所有优化的地基——地基歪了，simplify 和 FP16 都白搭。

§2　onnxsim：把“一团乱麻”精简成干净图

开场类比：把绕远的路线图捋直 导出时，PyTorch 里一句简单的 x.view(...) 常常会被 trace 成一长串 Shape→Gather→Unsqueeze→Concat→Reshape 的“绕路”节点——逻辑没错，但图里塞满了 “算来算去其实是个常数”的废动作。onnxsim 就是把这些绕路捋直：能在编译期算出来的， 就提前算成一个常量；能合并的，就并成一步。

它做的三件事

  • 常量折叠（constant folding）：把只依赖常量的子图，提前算出结果替换成一个常量。上面那串 Shape/Gather 全是基于固定形状的运算，直接折成一个常量。
  • 形状推理 + 去冗余：推断每条边的形状，删掉 Identity、无用的 Cast、空操作等。
  • 结构化简：让图更紧凑，给后续的算子融合、FP16 转换、引擎编译扫清障碍。
PYTHON / CLI · 图简化
# 方式一：命令行，最省事
#   onnxsim model.onnx model_sim.onnx

# 方式二：Python API
import onnx
from onnxsim import simplify

model = onnx.load("model.onnx")
model_sim, ok = simplify(model)        # ok=True 表示简化后与原图数值校验通过
assert ok, "简化后一致性校验失败"
onnx.save(model_sim, "model_sim.onnx")
易错点 ① 动态维度 + 简化有时会让形状推理失败——可先固定输入尺寸再 simplify，或用 --overwrite-input-shape； ② simplify 返回的 ok 一定要 assert，False 说明简化前后对不上，别盲目使用； ③ onnxsim 只“化简图”，不改数值精度——它和下一节的 FP16 是两件正交的事，通常先 simplify 再转 FP16。

§3　onnxconverter-common：FP16 转换，体积显存直接砍半

开场类比：账本从“分”改记到“角” 模型权重默认是 FP32（32 位浮点，每个数 4 字节）。但神经网络的权重根本不需要那么高的精度—— 就像记一笔几百块的账，记到“分”纯属浪费。FP16（16 位浮点，每个数 2 字节）把记账精度降一档， 每个数省一半空间。模型有上亿个数，省下来的体积和显存就非常可观，而预测结果几乎看不出差别。

为什么 FP16 正好省一半：看 bit 怎么分

浮点数由三段组成：符号位 sign + 指数位 exponent + 尾数位 mantissa。一个浮点数的值是：

浮点数取值公式 \[ \text{value} = (-1)^{\text{sign}} \times \left(1 + \text{mantissa}\right) \times 2^{\,\text{exponent}-\text{bias}} \] 其中指数位决定能表示多大/多小的范围，尾数位决定精度（有效数字）。

再补一个深度学习里常见的“第三种”：BF16。它把 16 位的预算换一种切法——保留 FP32 的 8 位指数（范围一样大），只砍尾数：

S·1
指数 8 位
尾数 23 位
FP32：范围大、精度高，但 4 字节。
S·1
指数 5
尾数 10 位
FP16：尾数多→精度较好，但指数只有 5 位→范围小，易溢出/下溢（最大约 ±65504）。
S·1
指数 8 位
尾数 7
BF16：指数和 FP32 一样大→范围不缩、几乎不溢出，代价是精度（尾数）更糙。训练更爱用它。
一句话记住三者权衡 同样 16 位，FP16 把预算给了精度（尾数 10 位），BF16 把预算给了范围（指数 8 位）。 部署推理常用 FP16（够准、又被各推理引擎/国产卡很好支持）；训练里为防梯度溢出常用 BF16。 无论哪种，相比 FP32 都是每个数 4 字节 → 2 字节，体积/显存约减半。

用 onnxconverter-common 一行转 FP16

PYTHON · FP16 转换
import onnx
from onnxconverter_common import float16

model = onnx.load("model_sim.onnx")         # 先 simplify 再转，效果最好
model_fp16 = float16.convert_float_to_float16(
    model,
    keep_io_types=True,                  # 输入/输出仍保持 FP32，外部调用无感、更稳
    min_positive_val=1e-7,             # 太小的正数下溢→截断到该下界
    max_finite_val=1e4,                # 太大的数溢出→截断到该上界（防 inf）
    op_block_list=["ReduceMean", "Softmax"],  # 对精度敏感的算子保持 FP32
)
onnx.save(model_fp16, "model_fp16.onnx")
几个关键参数的含义
  • keep_io_types=True：图内部用 FP16 算，但对外接口仍是 FP32，省去调用方改数据类型的麻烦，也避免接口处精度问题。
  • min_positive_val / max_finite_val：FP16 范围窄，超出会变 inf/0。这两个参数把数值钳制在安全区间（注意 0/nan/inf 本身保持不变）。
  • op_block_list / node_block_list：把对精度敏感的算子/节点保留为 FP16 之外（FP32）——这是控精度的核心旋钮，下面专门讲。

FP16 的“翻车点”与兜底：auto_mixed_precision

哪些算子容易在 FP16 下出问题 ① 累加/归一类：ReduceMean / ReduceSum / LayerNorm / Softmax——大量数相加，FP16 尾数少会累积误差，甚至出 NaN/全零； ② 数值跨度大的：含很大或很小常量的算子，FP16 直接溢出/下溢； ③ 对策：把它们丢进 op_block_list 保持 FP32，其余照转——这就是混合精度。

如果你不想手工挑哪些算子要 block，可以用自动混合精度：它拿一组校验输入，自动找出“一转就掉精度”的算子并保留为 FP32，在“尽量省”和“别掉精度”之间自动权衡：

PYTHON · 自动混合精度兜底
from onnxconverter_common import auto_mixed_precision
import numpy as np

feed = {"input": np.random.randn(1,3,224,224).astype(np.float32)}
# validate: 一个判定函数，比较 FP32 与混合精度输出是否在容忍范围内
model_mixed = auto_mixed_precision.auto_convert_mixed_precision(
    model, feed, validate_fn=lambda r0, r1: np.allclose(r0[0], r1[0], atol=1e-2),
)

§4　完整示例：小 CNN 走一遍全流水线，量化对比收益

目标 把一个普通 CNN 走完 导出 → 简化 → FP16 三步，并测出每一步带来的体积变化、数值误差， 建立“这套操作到底省多少、掉多少精度”的直觉。这正是评分里 U（体积）与 W（精度别掉）的迷你预演。
PYTHON · 端到端流水线（可直接改成你的模型）
import torch, onnx, os, numpy as np
import onnxruntime as ort
from onnxsim import simplify
from onnxconverter_common import float16
import torchvision.models as tvm

def size_mb(p): return os.path.getsize(p) / 1e6

# 0) 准备模型与样例输入
model = tvm.resnet18(weights=None).eval()
dummy = torch.randn(1, 3, 224, 224)

# 1) 导出 ONNX（FP32 基线）
torch.onnx.export(model, dummy, "r18.onnx", opset_version=17,
                  input_names=["input"], output_names=["output"])

# 2) 图简化
m, ok = simplify(onnx.load("r18.onnx")); assert ok
onnx.save(m, "r18_sim.onnx")

# 3) FP16 转换
m16 = float16.convert_float_to_float16(onnx.load("r18_sim.onnx"), keep_io_types=True)
onnx.save(m16, "r18_fp16.onnx")

# 4) 对比体积
for p in ["r18.onnx", "r18_sim.onnx", "r18_fp16.onnx"]:
    print(f"{p:14s}  {size_mb(p):6.2f} MB")

# 5) 对比数值误差：FP32 基线 vs FP16
x = np.random.randn(1,3,224,224).astype(np.float32)
o32 = ort.InferenceSession("r18_sim.onnx").run(None, {"input": x})[0]
o16 = ort.InferenceSession("r18_fp16.onnx").run(None, {"input": x})[0]
print("最大绝对误差:", np.abs(o32 - o16).max())
你会看到的典型结果（量级感）
  • 体积：FP32 约 45 MB → FP16 约 23 MB，≈ 减半（simplify 这步体积变化很小，它主要清理结构、为后续提速铺路）；
  • 显存：实际推理时权重 + 激活也大致随之减半（显存还取决于激活与中间缓冲，FP16 同样受益）；
  • 数值误差：最大绝对误差通常在 1e-2 ~ 1e-3 量级，分类 top-1 几乎不变——这就是“几乎不掉精度换一半空间”。
把误差量到“分数”上的提醒 赛题的 W 看的是 RMSE/ACC 相对原模型有没有掉。FP16 在大多数视觉/回归任务里误差极小，但气象这种数值范围大、对小量级敏感的任务， 务必在验证集上实测 RMSE/ACC，对累加/归一类算子按需 block，必要时上 BF16 或自动混合精度。别凭“一般不掉”想当然。

§5　易错点汇总 · 一图流 · 赛题落点

⚠️ 易错点汇总

  • 忘了 model.eval()：BN/Dropout 处于训练态，导出的图行为和推理不一致。
  • dummy 输入选错路径：模型里有 if 分支时，trace 只记录走过的那条；用代表性输入，必要时改用 dynamo/脚本化。
  • opset 太低：新算子导不出或被拆成一堆碎算子；建议 opset ≥ 13（常用 17）。
  • simplify 的 ok 没校验：返回 False 还硬用，等于埋雷。
  • FP16 直接全转：不 block 累加/归一算子，容易出 NaN/精度塌方——先小批验证再全量。
  • 只看体积不看显存：评分 U 里显存单独占 20 分，要实测推理峰值显存，而不仅是文件大小。
🎯 一图流小结：三件套各管一件事 onnx（导出）：PyTorch 私房菜 → 标准菜谱（计算图），是优化的舞台。
onnxsim（简化）：捋直绕路、折叠常量、去冗余 —— 改“结构”，不改数值，为提速与编译铺路。
onnxconverter-common（FP16）：把记账单位 4 字节→2 字节 —— 改“精度位宽”，体积/显存约减半。
顺序固定：导出 → simplify → FP16（敏感算子 block）→ 交给引擎编译。
🔗 赛题落点：这一套在 pangu-weather + 国产加速卡上怎么拿分
  • U（轻量化 40 分）：FP16 把权重文件与显存占用直接砍半，是性价比最高的第一刀；onnxsim 顺手清掉导出冗余，让“模型大小”统计更干净。
  • V（推理时长 20 分）：简化后的干净图 + FP16，交给国产卡的图推理引擎 MIGraphX（类 TensorRT）做算子融合与编译，延迟明显下降。
  • W（精度 40 分）：气象数值敏感，必须在验证集实测 RMSE/ACC；把 ReduceMean/LayerNorm/Softmax 等按需 block，或用自动混合精度守住精度。
  • 组合预告：FP16 是“压精度位宽”，下一讲的剪枝是“去结构冗余”，第三讲的张量分解是“压低秩冗余”——三招正交，可叠加。
