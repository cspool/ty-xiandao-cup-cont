GPU 推理优化系列 · 第三集

把带宽喂满：带宽利用率优化与对应的算子优化

上一集我们搞清了「decode 是带宽墙」。这一集只回答一个问题：既然撞了带宽墙，怎么把这堵墙的带宽真正榨满？——为什么利用率上不去，以及每一种「提利用率」的算子优化手法背后的原理。只讲思路与原理，附 LaTeX、伪代码、内联示意图、HIP 适配与 GitHub。

0 · 这一集要解决的问题

第二集的结论：自回归推理的 decode 阶段是 memory-bound——每生成一个 token，都要把整个模型权重从显存（HBM）读一遍，计算极少。它的速度下界是：

$$ T_{\text{decode/token}} \;\gtrsim\; \frac{\text{每 token 需读取的权重字节数}}{B_{\text{HBM}}} $$

这个下界由硬件带宽 $B_{\text{HBM}}$ 决定，动不了。但现实里，kernel 往往达不到这个下界——实测时间比理论下界大不少。差在哪？差在 带宽利用率。这一集就是讲怎么把利用率从「中位」推向「峰值」。

1 · 什么叫「把带宽喂满」

定义带宽利用率 $\eta$：实际达成的访存带宽，占硬件峰值带宽的比例。

$$ \eta=\frac{B_{\text{achieved}}}{B_{\text{peak}}}=\frac{\text{该 kernel 读写字节数}\,/\,\text{实测耗时}}{B_{\text{peak}}} $$

在 Roofline 图上，memory-bound 的 kernel 落在斜坡上。「优化」对它而言不是往右（提算术强度），而是把点沿斜坡往上顶，贴近那条带宽斜线——也就是提高 $\eta$。

图 1 · memory-bound 的优化 = 把实测点沿带宽斜线往上顶（提利用率 η），而不是往右
关键认知判断「还有没有优化空间」就看 $\eta$：$\eta$ 离 1 还远 → 有肉；$\eta\approx1$（贴斜线）→ 撞墙到顶。这比看「GPU 忙不忙」靠谱得多。

2 · 为什么带宽利用率上不去（四个元凶）

同样读 50 GB 权重，有人 60% 利用率、有人 95%，差距全在这四点：

① 访存延迟没藏住

一条 load 指令发出，到数据真正回来，有几百个时钟周期延迟。如果计算单元发完 load 就干等，SM 大量空转——「忙着等内存」。

② 访存不合并（uncoalesced）

一个 wavefront/warp 里的线程若读的是不连续地址，硬件要拆成多次内存事务，每次只搬到一点有效数据，带宽白白浪费。

③ kernel 太碎

把一串小算子拆成很多个 kernel，每个都要把中间结果写回 HBM、下一个再读回来，凭空多出大量往返；外加每次 kernel 启动/图回放的固定开销。

④ 库的 kernel 不对路

通用 BLAS 库为「大方阵」优化。推理 decode 是 M=1 的瘦长矩阵×向量，套大矩阵 kernel 会严重浪费，利用率自然低。
对应解法（下一节逐个讲）① → 软件流水/双缓冲；② → 合并 + 对齐访存；③ → 算子融合；④ → 选对/手写专用 kernel。

3 · 提带宽利用率的算子优化（逐个拆原理）

A内存合并访问（coalesced access）

让一个 wavefront 内相邻线程读相邻地址，凑满一次宽内存事务（典型 128 字节），把每次搬运的有效率拉满。再配合宽 load（一次读 128-bit，如 float4/dwordx4）和地址对齐，进一步减少事务数。

图 2 · 合并访存把「搬运有效率」从碎片拉到接近 100%

B软件流水 / 双缓冲（藏访存延迟）—— memory-bound 的头号杠杆

核心思想：在算当前这块数据时，提前把下一块的 load 发出去。用两组缓冲（寄存器或 LDS）轮转，让 global load 的长延迟和计算重叠，而不是串行等待。理想情况下每一拍的有效耗时从「读+算」变成「读、算的较大者」：

$$ T_{\text{串行}}=\sum (T^{(i)}_{\text{load}}+T^{(i)}_{\text{compute}})\;\;\Longrightarrow\;\; T_{\text{流水}}\approx T_{\text{load}}^{(0)}+\sum \max\!\big(T^{(i)}_{\text{load}},T^{(i)}_{\text{compute}}\big) $$

对纯 memory-bound（compute 极小）的 decode，效果就是把「等内存的空转」填掉，利用率显著上抬。

# 伪代码：双缓冲软件流水（A/B 两组缓冲轮转）
prefetch load block[0] → buf[0]          # 先发第一块的 load
for i in 0 .. N-1:
    if i+1 < N:
        prefetch load block[i+1] → buf[(i+1)%2]  # 发下一块(异步,不等)
    wait(buf[i%2])                            # 只等当前这块就绪
    compute(buf[i%2])  // 与上面的 load 在硬件上重叠执行
图 3 · 双缓冲把计算藏进 load 的「影子」里，时间退化为「读带宽」本身 = 喂满 HBM

C算子融合（减 HBM 往返 + 减 kernel 数）

把数据流上连续的几个小算子合进一个 kernel：中间结果留在寄存器/LDS、不落回 HBM，下一步直接用。既省掉中间张量的写出+读回（带宽），又减少 kernel 个数（启动/图回放的固定 gap）。

融合前

norm → 写 HBM → 读 HBM → 激活 → 写 HBM → 读 HBM → 残差…
每个箭头都是一次 HBM 往返。

融合后

一个 kernel 内：norm·激活·残差全在寄存器里串起来，只读一次输入、只写一次输出。
经典范例注意力的 tiling + online-softmax（FlashAttention 思路），就是把「QKᵀ→softmax→×V」融成一趟、避免把巨大的注意力矩阵写回 HBM——本质是用融合消灭中间结果的带宽。
注意融合不是越多越好：受限于寄存器/LDS 容量；过度融合会降 occupancy。而且很多框架的编译器（如 inductor 的 combo kernels）已自动融了一批小算子，手融前先看它融了没，别做无用功。

D选对 / 手写专用 kernel（瘦长 GEMV）

推理 decode 的线性层是 $M=1$ 的矩阵×向量（GEMV）：权重矩阵很大、激活只有一列。通用 GEMM 库按「大方阵 tile」切，处理 $M=1$ 时一个 tile 的绝大部分行是浪费，带宽利用率上不去。对策是走专门的 GEMV 路径，或针对该形状手写 kernel：一个 wavefront 负责若干输出行、流式读权重、wave 内规约、配合上面的合并访存 + 双缓冲。

# 伪代码：M=1 GEMV，一个 wave 算若干输出行（要点而非实现）
each wavefront → 负责输出向量的 ROWS 行:
    acc[ROWS] = 0                      # 累加器常驻寄存器
    for 每个 K 分块 (配双缓冲预取):
        w = load 权重该分块   # 合并 + 宽 load + non-temporal
        x = 共享的激活分块       # 放 LDS/寄存器，全 wave 复用
        acc += dot(w, x)               # 乘加
    acc = wave_reduce(acc)             # wave 内规约(shuffle)
    write acc → 输出
判断要不要自己写先 A/B：拿库默认 kernel 和候选实现各测一遍达成带宽。只有当库在这个形状上明显没喂满（$\eta$ 低）时，手写才有意义；库已接近峰值就别硬刚。

E流式读 + 数据复用（少读、绕 cache）

  • non-temporal / streaming load：权重在 decode 里只读一次、不再复用，让它绕过 cache，避免污染、把缓存留给真正会重用的数据。
  • 寄存器 / LDS 驻留复用：能复用的数据只从 HBM 读一次，之后留在片上反复用。例如分组注意力（GQA）里多个 query head 共享同一份 K/V——读一次供多头复用，把这部分的有效带宽需求直接除以分组数。
$$ \text{有效读量}_{\text{复用后}}=\frac{\text{有效读量}_{\text{朴素}}}{\text{复用倍数}}\quad\Rightarrow\quad \text{同样带宽下耗时下降} $$

4 · 怎么判断「到顶了」

有了带宽利用率这把尺子，判断就很干脆：

# 用「纯读探针」标定该卡真实可持续带宽，再判断 kernel 到没到墙
B_probe = 跑一个只读不算的 kernel 测得的 GB/s   # 该卡真实上限(比规格可信)
T_floor = kernel 读写字节数 / B_probe              # 理论下界
eta     = T_floor / 实测耗时
if eta > 0.9: "贴峰值,撞带宽墙,这条路到顶 → 换思路"
elif eta < 0.7: "没喂满 → 上 合并/双缓冲/融合/专用kernel"

当所有热点 kernel 的 $\eta$ 都贴近峰值、且能融的中间结果都融掉了，就说明在「合规、不改精度」的前提下，带宽这条路已经到物理天花板——剩下的只能去碰算法层（而那往往受赛题规则约束）。

5 · 在 HIP / 国产加速卡（DCU）上怎么落地

上面的原理硬件无关——带宽墙是 HBM 的物理属性。换到 ROCm/HIP 体系（国产 DCU、AMD），只是工具和一些细节不同：

能力 / 概念        CUDA(NVIDIA)        HIP/ROCm(AMD · 海光DCU)
监控               nvidia-smi          rocm-smi / hy-smi
kernel profiler    Nsight Compute      rocprof / omniperf(可出 Roofline)
设备/架构信息      deviceQuery         rocminfo
线程束             warp = 32           wavefront = 64 (gfx9)
片上暂存           shared memory       LDS
宽 load            float4 / ld.128     dwordx4 / global_load
BLAS               cuBLAS/cuBLASLt     rocBLAS / hipBLASLt
低精度矩阵单元     Tensor Core         Matrix Core (MFMA/mmac)
  • 测利用率：用 omniperf 出 Roofline、看 memory throughput；峰值带宽用「纯读探针」实测（同上）。
  • wavefront = 64：跨线程规约、__shfl 宽度是 64，从 CUDA 搬 kernel 时最易踩错。
  • 双缓冲/合并/non-temporal：思路完全通用，HIP 同样支持异步 load、宽 load、流式提示。
  • 库要 A/B：rocBLAS / hipBLASLt 在 $M=1$ 这种瘦长形状上的默认选择不一定最优，务必两个都测、选达成带宽高的。
  • inline asm 与矩阵核：HIP 也能内联汇编调矩阵指令，但不同代/厂商的指令助记符可能不同（编译器内建函数在定制工具链里可能缺，需用对应汇编指令）——国产卡移植的常见坑。
  • gfx9 部署：容器映射 /dev/kfd /dev/dri、设对 HSA_OVERRIDE_GFX_VERSION；为新架构优化的算子库在老 gfx 上可能反而更慢，按实测开关。
一句话方法论一字不变（Roofline 判墙、盯带宽利用率），只换工具（omniperf）、注意 wave64 与库后端选择。「把带宽喂满」在国产卡上同样是 decode 优化的主战场。

7 · 小结

  • 撞了带宽墙，优化目标就是提带宽利用率 $\eta$——把 Roofline 上的点沿斜线往上顶到贴峰值。
  • 利用率上不去的四个元凶：延迟没藏住、访存不合并、kernel 太碎、库 kernel 不对路。
  • 对应五招：合并访存、双缓冲软件流水（头号杠杆）、算子融合、专用 GEMV kernel、流式读+数据复用。
  • 用纯读探针 + 利用率判断到没到顶；$\eta\to$ 峰值即合规手段下的物理天花板。
  • 换 HIP / 国产卡，原理不变，只换工具（omniperf）、注意 wave64 与库 A/B。
