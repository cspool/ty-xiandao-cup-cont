## 2026年全国大学生计算机系统能力大赛 智能计算创新设计赛(先导杯)

# 基于国产加速卡的Qwen大模型推理服务优化 赛题技术方案

## 一、赛题总体介绍

- 第1条 随着大语言模型进入工程部署落地阶段,推理服务性能已成为影响部署成本、 服务稳定性和用户体验的关键因素。尤其在长上下文与混合长度场景下,随 着输入序列长度的增加,单请求的资源需求急剧上升。KV Cache 占用随上下 文长度线性膨胀,HBM 带宽压力与显存碎片问题突出,容易导致首 token 时 延(TTFT)上升、输出吞吐下降。当前统一的 vLLM 推理框架具备基础的处理 能力,但在长上下文单请求负载下,默认机制仍难以充分挖掘国产加速卡 (本赛题采用 DCU)的计算与带宽潜力,兼顾吞吐、时延与显存占用。
- 第3条 本赛题的性能优化聚焦于不同输入长度的单请求在线推理服务。参赛者需结合 Qwen3.5-27B 模型特点、长上下文输入特征和 DCU 硬件特性,采取针对性优 化策略,包括但不限于:KV Cache 与显存管理优化、执行路径与算子优化, 以及在允许边界内的精度无损压缩方案。
- 第2条 参赛者需在统一模型权重、统一容器镜像、统一 vLLM 0.18.1 推理框架和统一 服务接口条件下开展评测。优化方案的核心目标是:在 TTFT P99 和 TPOT P99 均满足规定 SLA 门限的约束前提下,最大化输出吞吐量,并改善显存使 用效率与服务稳定性。SLA 为硬性约束,违反则该档位吞吐量得分直接清零。
- 第4条 镜像预装内容固定为 Python 3.10.12、指定版本 PyTorch 2.10.0、vLLM 0.18.1、 固定版本 transformers 5.5.0、Qwen3.5-27B 官方 tokenizer、官方 chat template 和官方 bf16 原始权重以及官方基准测试工具。选手可叠加自定义 层,包括安装 Python 包、编译 custom kernel,但需提交改动说明。最终评 测仅在统一容器环境内执行,宿主机不注入任何额外库,评测期间禁止从外

## 二、初赛要求与评分标准

第5条 本赛题初赛聚焦真实场景下的在线推理服务效率优化,主要挑战维度如下:

1. KV Cache 与显存管理优化

高效块管理与碎片化控制:设计更高效的 PagedAttention 块分配与回收机制, 针对长上下文请求优化块粒度与对齐策略,降低单请求内的显存碎片率与 HBM 访问 开销。

#### 2. 框架与算子优化

- (1)Decode 阶段算子优化:针对生成阶段的 Attention 算子、Linear 算子, 结合 DCU 的显存带宽与计算特性,进行算子融合与内核优化,降低每 Token 生成耗时(TPOT)。
- (2)执行路径优化:减少 Python 层调度开销、降低 kernel launch 延迟,优 化调度逻辑与内存拷贝路径,缩短首 Token 时延(TTFT)。

#### 第6条 本赛题初赛建议的优化内容包括:

- 1. Decode 阶段调度进行深度定制。
- 2. 允许对 KV Cache 的分配机制、显存预算控制和块管理策略进行优化。
- 3. 允许结合 DCU 的显存容量、带宽、片上缓存层级、并行执行特征和数据搬运 开销,对资源组织调度方式进行适配优化。

#### 第7条 本赛题初赛禁止的行为包括:

- 1. 不得通过修改采样参数(如强行调低 Max Tokens)、截断输入、跳过长样本、 过滤困难样本、不得跳过层等方式规避真实推理开销。
- 2. 不得通过预缓存测试集与答案、预生成中间结果等针对性优化手段响应请求。
- 3. 不得绕开统一服务接口、评测流程与资源统计路径,不得引入题目规定外的 辅助模型(如外挂自行训练的小模型用于投机采样等)。
- (1)严禁修改模型结构,替换模型权重、改变模型推理语义或输出口径。不得 通过后训练、蒸馏、微调等方式修改模型。
  - (2)量化操作边界:禁止在权重加载前、权重加载后、服务初始化阶段或正式 推理前,对模型权重进行持久化量化、结构化剪枝、权重重排压缩、模型图 重构、模型格式转换或生成可复用量化权重缓存。(允许推理过程中的非持

久化、算子级低精度计算优化,例如激活值动态量化、KV Cache 量化、 kernel 内部临时类型转换、低精度矩阵乘法、Attention 内核优化等。)

- (3)模型剪枝边界:禁止任何形式的结构化剪枝、非结构化剪枝、动态通道跳 过、动态层跳过、注意力头裁剪、token pruning、early-exit 等改变模型 有效计算结构或输出语义的行为。
- (4)推理过程中不得生成可供后续样本复用的新模型文件、量化权重文件、剪 枝权重文件或压缩权重缓存。

#### 第8条 初赛主要扣分与失效点如下:

- 1. SLA 熔断:吞吐提升明显,但 TTFT、TPOT(每个输出 Token 的平均耗时) 的 P99 时延严重超标,或服务完成率下降超过 1%,该档位吞吐得分将直接 清零。
- 2. 方案与显存、带宽、访存等物理约束脱节,或纯粹针对特定评测数据集写死 的 If-Else 过拟合规则。

#### 3. 评测相关指标定义

为避免统计口径争议,评测时所有核心指标遵循以下严格定义:

(1)性能评测:统一使用 vllm bench serve 完成,用于对参赛者部署的推理服 务进行压测,并统计服务性能相关指标,包括输出吞吐量(Throughput)、

- 首 Token 时延(TTFT)、TPOT。
- (2)精度评测:统一使用 OpenCompass 完成,用于对问答、摘要、检索、聚合 等任务的输出结果进行精度评估。精度评测结果用于精度扣分计算,不参与 吞吐量或时延指标统计。
- (3)吞吐量(Throughput): 严格定义为输出吞吐量(Output Tokens / Second)。计算逻辑为:所有完成请求的生成 Token 总数 /(最后一个请求 完成的时间点 - 第一个请求发出的时间点)。输入的 Prompt Token 不计入 吞吐量分子。
- (4)TTFT(Time To First Token,首 Token 时延): 采用客户端端到端计时。
- · 起算点:评测客户端向推理服务 API 发出 HTTP 请求的绝对时间戳。
- · 终止点:评测客户端收到流式返回第一个生成 Token 的绝对时间戳。
- · 包含内容:TTFT 明确包含 Tokenizer 编码时间以及首 Token 的 Prefill 计算时间。初赛题并发数固定为 1,引擎内无排队等待时间。
- (5)TPOT(Time Per Output Token,每 Token 生成耗时): 对每条请求,定 义为从收到第一个 Token 到收到最后一个 Token 的时间差,除以(输出 Token 数 - 1)。将全部请求的 TPOT 值汇入全局池后取 P99 分位值(TPOT P99)。输出 Token 数为 1 的请求不参与 TPOT 统计。
- (6)P99 统计维度:
- · TTFT P99:按请求维度统计,取全部请求首字时延的 P99 分位值,各档位分 别计算,各档独立判断是否达标。
- · TPOT P99:所有请求的 TPOT 值汇总后取 P99 分位值,全局统一(不分档), 适用于所有档位的 SLA 判断。
- ( 7 ) 官 方 Baseline 核 心 参 数 示 例 ( 评 测 时 严 格 锁 定 ) : vllm serve Qwen/Qwen3.5-27B --max-model-len 32768 。 以 下 参 数 严 格 锁 定 : max\_tokens、temperature=0、max-num-seqs。
- (8)不允许调整的参数(所有影响任务定义、上下文范围和输出行为的参数严 格锁定)。以下按类别列出不可调整项;若与本说明存在差异,以评测脚本 与实际评测命令的锁定为准。
- 模型 与分词: 模型权重 (model)、tokenizer 、tokenizer-mode、 chat template。
- 上下文与批处理:--max-model-len、--max-num-seqs、--max-num-batched-

tokens 及其它 batch scheduler 相关参数(包括不允许 vLLM 内部 batch scheduler 相关代码修改)。

- 禁止能力:任何形式的投机解码,包括但不限于 draft model、MTP、多头预 测、外挂小模型、自训练预测器、基于中间层的 early-exit draft、预生成 token 序列缓存等。参赛方案必须基于官方模型的标准自回归解码流程完成 输出生成。
- Bench 与结果:bench 的 percentile/percentiles 统计口径、结果保存选项、 结果解析脚本。
- 服 务 接 口 : --served-model-name 、 OpenAI API 路 径 、 请 求 / 响 应 格 式 (host/port/路由由平台固定)。
- 4. 输出吞吐量(总计 100 分)

吞吐量评分基于统一的、提前评测验证的官方基准线(Baseline)计算相对提升, 杜绝调参套利。

在 SLA 门限通过的前提下,按三档输入长度分别计算吞吐提升得分,三档权重 分别为 20%、50%、30%,各档满分分别为 20 分、50 分、30 分,合计 100 分。

单档相对提升率 = (参赛方案该档吞吐量 − Baseline该档吞吐量)/Baseline该档吞吐量

各档得分 <sup>=</sup> 该档满分 <sup>×</sup> (60% <sup>+</sup> 40% <sup>×</sup> (1 <sup>−</sup> <sup>e</sup>−1.3×单档相对提升率))

5. 时延 SLA 熔断机制(硬性约束,不另计分值)

服务可用性通过首 Token 时延(TTFT)和每 Token 生成耗时(TPOT)联合评估,仅作 为硬性约束,不另计得分。

#### SLA 门限:

- · TTFT P99(各档分别)≤ Baseline TTFT P99(对应档)× 1.5
- · TPOT P99(全局)≤ Baseline TPOT P99(全局)× 1.5

熔断机制: 各不同上下文长度档位的吞吐测试中,一旦上述任一 SLA 指标未能达标, 该档位对应的吞吐量得分直接清零。

6. 约束条件:模型精度

初赛精度评估与吞吐评测使用同一套任务类型与评分口径,统一要求如下:

- · 评测时 Temperature= 0.0;
- · 所有任务均与官方 Baseline 输出结果或标准答案进行对比;

对四类任务分别计算精度结果,并统计相对 Baseline 的性能下降幅度,记为 Δ,根

据下表得到各任务对应精度系数ki,取四类任务系数均值作为最终精度系数。

| 精度下降幅度 Δ                     | 精度系数 k      |  |
|------------------------------|-------------|--|
| $\Delta \leqslant 1.0\%$     | 1.00        |  |
| $1.0\% < \Delta \leq 2.0\%$  | 0. 97       |  |
| $2.0\% < \Delta \leq 3.0\%$  | 0.94        |  |
| $3.0\% < \Delta \leq 5.0\%$  | 0.90        |  |
| $5.0\% < \Delta \leq 10.0\%$ | 0. 85       |  |
| Δ > 10%                      | 0 (该类任务不计分) |  |

 $\Delta$  = (Baseline精度-参赛方案精度)/Baseline精度×100% 最终得分=吞吐量得分× $\frac{1}{4}$ × $\sum_{i=1}^{4}k_i$
